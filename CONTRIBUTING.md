# Contributing to Git Helper

Thanks for wanting to help! Git Helper is a friendly visual layer over Azure
DevOps Repos and local Git for developers who don't live in the CLI. This
document explains how the project is organised, how changes flow from your
keyboard to a release, and what we look for in a contribution.

## TL;DR

1. Fork the repo (or create a branch if you're a collaborator).
2. Branch from **`develop`** — never from `main`.
3. Make your change, run `npm run typecheck` and `npm run build`.
4. Open a **pull request into `develop`**. CI must pass.
5. A maintainer merges it into `develop` → a **preview deployment** is created
   automatically → the maintainer verifies the preview and promotes
   `develop` → `main` when it's good. `main` is what users get.

## Branch model & preview links

| Branch | Purpose |
|---|---|
| `main` | Production. The hosted app deploys from here, and releases (`v*` tags) build the desktop installers from here. **Never commit to it directly.** |
| `develop` | Integration branch. All contributions land here first. |
| `feature/…`, `fix/…` | Your working branches, branched off `develop`. |

**Preview deployments:** the repo is connected to Vercel, so every push to
`develop` (and authorized pull requests) automatically gets a **preview
deployment** — a unique URL with the app built from that exact commit. You'll
find the link on the commit / PR (the Vercel bot posts it, and it appears
under the repo's *Deployments* section). Maintainers use this preview to
verify a change in a real browser before promoting anything to `main`.
Pull requests from forks get a preview after a maintainer authorizes the
build — this is a security measure, not a snub.

Release flow (maintainers): merge `develop` → `main`, then tag `vX.Y.Z`.
The tag triggers the release workflow, which builds the Windows and macOS
installers and publishes them on GitHub Releases.

## Getting the dev environment running

Prereqs: **Node 18+** and a **`git`** binary on your PATH.

```bash
npm install
cp .env.example .env      # set SESSION_SECRET to any random string
npm run dev               # http://localhost:5173
```

The Express API is mounted as Vite middleware, so one command runs
everything. To try Azure DevOps features you'll need an Azure DevOps
organization and a PAT (Code → Read & write) — local-Git features work with
any repo folder on your machine, no Azure needed.

Desktop app (Electron):

```bash
cd desktop && npm install && npm start   # builds the web app + server bundle, then launches
```

## Project layout (30-second tour)

```
src/            React app (TypeScript, Tailwind, TanStack Query)
  pages/        One file per screen (Status, Changes, Commit, Recovery, …)
  components/   Shared UI (ConfirmDialog, NetworkToolbar, AssistantPanel, …)
  api/client.ts All fetch calls + shared types — the only place that talks HTTP
server/         Express API
  localGit.ts   The local-git engine (every git invocation lives here)
  azdo.ts       Azure DevOps REST client
  routes/       HTTP routes (local.ts, branches.ts, pullRequests.ts, …)
  mcp.ts        MCP server exposed to Claude Desktop
desktop/        Electron shell + packaging (electron-builder)
scripts/        esbuild bundling for Vercel/desktop/MCP + PDF guide generator
docs/           User guide (markdown + generated PDF)
```

## What we look for in a PR

- **Typecheck and build pass**: `npm run typecheck` and `npm run build` are
  the same gates CI runs.
- **Plain language in the UI.** The whole point of this app is that someone
  who doesn't know git can use it. Every new action needs a "what will
  happen / what it affects" explanation, and destructive actions must go
  through `ConfirmDialog` (typed confirmation for anything unrecoverable).
- **Safety rules are non-negotiable:**
  - Never rewrite pushed history silently; never force-push on a user's behalf.
  - Never run a destructive git command without an explicit confirm.
  - PATs and API keys stay server-side — never in browser storage, git
    config, remote URLs, or logs.
- **Match the existing style.** Look at a neighbouring page/route and write
  code that looks like it. Comments explain *why*, not *what*.
- **Small PRs merge fast.** One feature or fix per PR.
- If you change user-visible behavior, update `docs/USER-GUIDE.md` in the
  same PR (the PDF is regenerated by maintainers with
  `node scripts/build-guide-pdf.mjs`).

## Reporting bugs / proposing features

Use the issue templates. For bugs, include your platform (web / desktop,
Windows / macOS), the app version, and what the History/Status screens
showed. Screenshots help enormously. **Never paste a PAT, API key, or
`claude_desktop_config.json` contents into an issue.**
